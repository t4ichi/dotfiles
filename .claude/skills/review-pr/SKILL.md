PRのコード差分をレビューし、改善点や問題点を指摘してください。

引数としてPR番号またはURLが渡された場合はそのPRを、渡されない場合は現在のブランチに紐づくPRをレビュー対象とします。

引数: $ARGUMENTS

## 手順

### 1. レビュー対象のPR特定と差分取得

**引数がある場合:**

```bash
# PR番号またはURLから差分を取得
GIT_SSL_NO_VERIFY=1 gh pr diff <PR番号またはURL>

# PR情報の取得
GIT_SSL_NO_VERIFY=1 gh pr view <PR番号またはURL> --json title,body,baseRefName,headRefName,files
```

**引数がない場合:**

```bash
# 現在のブランチ名を取得
BRANCH=$(git branch --show-current)

# PRを検索
GIT_SSL_NO_VERIFY=1 gh pr list --head "$BRANCH" --json number,title,url

# PRが見つかった場合は差分を取得
GIT_SSL_NO_VERIFY=1 gh pr diff <PR番号>

# PRが見つからない場合は dev..HEAD の差分をレビュー対象にする
git diff dev..HEAD
```

### 2. 変更ファイルの把握

変更されたファイルの一覧を確認し、以下を把握する:

- 変更ファイルの種類（Go / TypeScript / OpenAPI / テスト / 設定 等）
- 変更の規模（追加行数・削除行数）
- 自動生成ファイル（`src/libs/orval/`、`*_gen.go`、`embedded_spec.gen.go`）は**レビュー対象外**とする

### 3. コードレビュー実施

差分の量が多い場合は、**Task tool（subagent）を並列で起動**してファイルごと・領域ごとにレビューする。

以下の観点でレビューする:

#### 必須チェック項目

| 観点 | 確認内容 |
|------|----------|
| バグ・ロジックエラー | nil参照、境界値、型ミスマッチ、競合状態、無限ループ等 |
| セキュリティ | SQLインジェクション、XSS、認証・認可の抜け、機密情報のハードコード |
| エラーハンドリング | エラーの握りつぶし、不適切なエラーメッセージ、リソースリーク |
| パフォーマンス | N+1クエリ、不要なループ内DB呼び出し、大量データの非効率な処理 |

#### 品質チェック項目

| 観点 | 確認内容 |
|------|----------|
| 命名・可読性 | 変数名・関数名が意図を表しているか、過度に複雑な処理がないか |
| 設計・構成 | クリーンアーキテクチャの層分離が守られているか、責務が適切か |
| テスト | 変更に対応するテストが追加・更新されているか |
| 一貫性 | 既存コードのパターン・規約に従っているか |

#### 言語固有の観点

**Go（backend/）:**
- Goイディオム（エラーハンドリング、defer、goroutineの安全性）
- GORMの使い方（クエリビルダの正確性、トランザクション管理）

**TypeScript（frontend/）:**
- 型安全性（any の使用、型アサーションの妥当性）
- React のベストプラクティス（hooks のルール、不要な再レンダリング）
- MUI コンポーネントの適切な使用

### 4. レビュー結果の報告

以下の形式で報告する:

```
## レビューサマリー

**対象PR:** #<PR番号> <タイトル>
**変更ファイル数:** N ファイル（+追加行 / -削除行）

## 指摘事項

### Must Fix（修正必須）
重大なバグ、セキュリティ問題など

| # | ファイル | 行 | 内容 | 理由 |
|---|---------|-----|------|------|
| 1 | path/to/file | L42 | 指摘内容 | 根拠・影響の説明 |

### Should Fix（修正推奨）
品質向上のための改善提案

| # | ファイル | 行 | 内容 | 理由 |
|---|---------|-----|------|------|
| 1 | path/to/file | L15 | 指摘内容 | 根拠・影響の説明 |

### Nit（軽微）
命名やスタイルなどの軽微な指摘

| # | ファイル | 行 | 内容 |
|---|---------|-----|------|
| 1 | path/to/file | L8 | 指摘内容 |

### Good（良い点）
設計やコードで特に良いと感じた点
```

指摘がないカテゴリは省略してよい。

## ルール

- 必ず日本語で報告する
- 自動生成ファイル（`src/libs/orval/`、`*_gen.go`、`embedded_spec.gen.go`）はレビュー対象外
- 指摘には必ず具体的なファイルパスと行番号を含める
- 改善案がある場合は具体的なコード例を提示する
- 推測ではなく、コードから読み取れる事実に基づいて指摘する
- Must Fix は本当に重大な問題のみに限定し、過度に厳しくしない
- 良い点も積極的に指摘する（レビューはダメ出しだけでなく、良い実装を認めることも重要）
