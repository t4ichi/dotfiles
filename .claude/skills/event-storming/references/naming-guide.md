# 命名の品質チェックガイド

イベントストーミングの本当の価値は「同じ言葉(ユビキタス言語)を作ること」にある。だから付箋の一語一語が、業務の実態を正しく・誤解なく映しているかを点検する。以下はレビュー(モードB)でも対話モデリング(モードA)でも使う判定基準。

判定の通奏低音はひとつ:**「この一言を見て、購読者(後続の処理・人)が起きたことを誤解なく理解できるか」「現場が実際にこの言葉を使っているか」**。

## 1. 業務イベントは過去形か

業務イベントは「すでに起きた、取り消せない確定事実」を表す。だから必ず過去形。

- ✅「注文が出荷された」「通話が接続された」
- ❌「注文を出荷する」→ これはまだ起きていない指示=**コマンド**

過去形を強制すると、参加者は「願望・手順」ではなく「実際に観測された出来事」だけを並べることになり、業務の実態がそのまま時系列に浮かぶ。命令形のイベントを見つけたら、コマンドとの混同を疑う。

## 2. CRUD・技術語の"におい"

「保存された」「更新された」「登録された(汎用)」「設定された」は、DBへの書き込みや画面操作という**仕組み**を述べがちで、業務の出来事を述べていないことがある。判定:

- ✅ 過去形である
- ❌ **業務上の重要な出来事か** — 「保存」は事業活動の出来事ではない
- ❌ **業務エキスパートがその言葉を使うか** — 現場は「保存した」ではなく「担当者を登録した/交代した」と言うはず

**赤信号(ほぼ常に直す):** 「情報が保存された」── 純粋な技術語。業務概念を持たない。
**黄信号(現場の言葉なら可):** 「次回コール日が設定された」── 「予定を決めた」という業務判断を含むので、現場が「設定する」と言うなら可。「予約する/予定する/アポを入れる」と言うなら直す。

「保存された」の裏には意味の異なる別イベントが隠れていることが多い:

```
✗ 企業担当者の情報が保存された          ← 技術(CRUD)・意味があいまい
○ 企業担当者が登録された                ← 新規。後続=ウェルカム通知
○ 企業担当者の連絡先が更新された        ← 変更
○ 企業の担当者が交代した                ← 引き継ぎ処理が走る
```

「登録」と「交代」では後続が変わる。一括りにすると業務上の分岐が消える。

## 3. コマンドとイベントの同居

「〜した結果〜」という複文は、1枚の付箋にコマンド(行為)とイベント(結果)を丸ごと書き込んでいるサイン。分ける。

```
✗ 架電した結果繋った
↓ 分解
[コマンド]  架電する        (アクター: オペレーター / 自動ダイヤラー)
[イベント]  通話が接続された         ← 正常系
[イベント]  通話が接続されなかった   ← 代替系
```

こうすると、後のステップでポリシー(再架電)や読み取りモデル(架電履歴)を素直に繋げられる。同居したままだと矢印を引く先がぼやける。

## 4. 能動と受動(態)

「割り当てた」(能動)と「割り当てられた」(受動)は**同じ事実の言い換え**。指す出来事は1つなので、**付箋は1枚・名前は1つ**。両方貼ると重複する。

混同しやすい2つの軸を区別する:

| 軸 | 選択肢 | 性質 |
|---|---|---|
| **時制の軸**(必須・概念の違い) | 過去形 ⇔ 命令形 | イベント vs コマンド。**区別する** |
| **態の軸**(ただの言い回し) | 受動 ⇔ 能動 | 同じイベントの書き方。**片方でよい** |

推奨は受動(対象中心)。購読者は「誰がやったか」より「どうなったか」に反応するため、行為者から切り離せる。行為者を明示したいときはイベント名に混ぜず、コマンド側のアクター付箋で表す。

## 5. 行為者で意味が変わる場合

行為者(アクター)違いで**その後の業務ロジックが変わる**なら、それはアクター違いではなく別イベント/別コマンド。付箋を分ける。判定基準は一貫して「後続(購読者・ポリシー)の反応が変わるか」。

- **後続が同じ** → イベントは1つ。行為者はアクター付箋か、イベントのデータ(payload)で持つ
- **後続が変わる** → 別イベントに割る。ただし名前は行為者ではなく**業務上の意味の違い**で付ける

```
✗ 管理者がキャンセルした / 顧客がキャンセルした   ← 行為者で命名(浅い)
○ 強制キャンセルされた   / 顧客都合でキャンセルされた ← 意味で命名(深い)
```

強いシグナル:業務エキスパートが2ケースを**別の言葉で呼んでいる**なら、ほぼ確実に別イベント。

## 6. 失敗・代替系もイベント

「繋らなかった」「却下された」「在庫が不足した」のような失敗・否定的結果も、業務上意味のある確定事実なら立派な業務イベント。何かがそれに反応する(再架電ポリシー、エラー通知、代替手配)からこそ書く。

さらに、反応が変わるなら失敗を種類で割る:「話中」「不在」「番号無効」は後続が違う(番号無効なら再架電しても無駄)ので別イベントにする価値がある。

## 7. 同じ言葉(ユビキタス言語)

最終判断基準はこれ。技術用語・抽象語ではなく、業務エキスパートが日常的に使う言葉に付箋を揃える。集約名・フィールド名・メソッド名・発行する業務イベント名まで、すべて区切られた文脈の同じ言葉で命名するのが理想(ソースコードにも一貫させる)。

レビュー時の問い:
- 「現場の人はこれを何と呼んでいますか?」
- 「この言葉、開発者だけが使っていませんか?」
- 「同じものを別の人が別の言葉で呼んでいませんか?」(相反するモデルの発見)
