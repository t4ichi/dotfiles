# ファシリテーションガイド

対話モデリング(モードA)で投げる問いの台本と、本番ワークショップ(モードC)の運営知識。

## 共通の心得

- **一問ずつ。** 一度に質問を畳みかけない。相手が一つ答えたら、その答えを起点に次を掘る。
- **語らせる。** あなたが図を埋めるのではなく、相手の言葉を引き出してそれを付箋に起こす。
- **食い違いを歓迎する。** 「人によって呼び方が違う」「順序が曖昧」が出たら、そこが宝。掘る。
- **過去形を促す。** 相手が命令形や手順で語ったら「それで何が起きましたか?(過去形で)」と返す。
- **問題点を拾う。** 全ステップを通じて、相手が漏らす不満・ボトルネック・「よく分からない」をピンクのダイヤとして書き留める。

## モードA:10ステップの問いの台本

### ステップ1:発散的に探索する
- 「この業務で起きる出来事を、順番は気にせず思いつくまま挙げてください。過去形で。」
- 「他には? もう出ないというところまで。」
- 重複・順序は今は無視。とにかく数を出す。

### ステップ2:時系列に並べる
- 「いちばんうまくいったとき(正常系)、最初に起きるのは何ですか? 次は?」
- 正常系が通ったら:「途中で失敗・分岐するとしたら、どこで、どうなりますか?」
- 「ここで2つに分かれますね。片方は『〜された』、もう片方は?」

### ステップ3:問題点を洗い出す
- 「この流れの中で、詰まりやすい・手作業で大変・誰も詳しくない箇所はどこですか?」
- 「さっき『面倒』とおっしゃったの、問題点として置いておきますね。」

### ステップ4:転換イベントを見つける
- 「フェーズや担当が切り替わる、節目になる出来事はどれですか?」
- 例:「ショッピングカートが初期化された」「注文が出荷された」「注文が返品された」
- 転換イベントの前後で別の文脈になりうる、という視点を共有する。

### ステップ5:コマンドを見つける
- 「『〜された』の直前に、誰か(または何か)が起こした操作は何ですか? 命令形で。」
- 「それを実行するのは誰ですか?(明らかなときだけアクターを付ける)」
- コマンドは失敗・却下されうる点を意識する。

### ステップ6:ポリシーを定義する
- 「このイベントが起きたら、人を介さず自動で次のコマンドが走りますか?」
- 「いつもですか? それとも条件付き?(例:VIP顧客のみ、3回失敗したら)」
- 条件はポリシーの付箋に明記。

### ステップ7:読み取りモデルを見つける
- 「このコマンドを実行する人は、判断のために何の画面・レポート・通知を見ますか?」
- 読み取りモデルはコマンドの前に置く。

### ステップ8:外部システムを追加する
- 「この業務領域の外にあって、コマンドを呼ぶ/通知を受けるシステムはありますか?」
- 例:CRM、決済代行、メール配信。
- 終了条件:全コマンドが「アクター/ポリシー/外部システム」のいずれかで発動する状態。

### ステップ9:集約を見つける
- 「関連の深いコマンドとイベントをまとめると、どんな『塊』になりますか?」
- 集約=コマンドを受けてイベントを生成する一貫性の境界。左にコマンド・右にイベント。

### ステップ10:区切られた文脈に分割する
- 「機能的に密接、またはポリシーで強く結びついた集約はどれとどれですか?」
- その塊が区切られた文脈(境界づけられたコンテキスト)の自然な候補。

### 二段構えの進め方(本の推奨)
導入時は、まず広い範囲でステップ1〜4だけ回して全体像と文脈のアウトラインを掴む。その後、特定した業務プロセスごとに全10ステップを回して各モデルを完成させる。

## モードC:本番ワークショップの運営

### 参加者
- 多様な立場が理想:エンジニア、業務エキスパート、プロダクトオーナー、テスター、UI/UXデザイナー、サポート担当。
- ただし大きくなりすぎない。全員が貢献できること。対面で10人、リモートは5人が目安。

### 必要なもの
- 広いモデリングスペース(壁一面の模造紙か大きなホワイトボード)、大量・多色の付箋、人数分のサインペン、ヘルシーなお菓子、椅子を撤去した広い部屋(議論より着席を防ぐ。立てない人向けに数脚は残す)。

### 進行の段取り(アジェンダ例・全体2〜4時間)
1. 導入(15分):これから何をするか、対象プロセスの概要、凡例(色分け)の説明。
2. ステップ1〜2(発散と時系列)に最も時間を割く。
3. 以降のステップは活力を見ながらタイムボックスで区切る。
4. 休憩は必要。ただし全員が戻るまで再開しない。再開時はモデルの現状確認から。

### ファシリテーションのコツ
- **凡例を常掲する。** 付箋の用途を書いた凡例を全員が見える位置に貼る。
- **変化に注目する。** 場の活力が落ちたら、問いで再活性化するか、次のステップへ進む頃合いかを見極める。
- **巻き込む。** 遠ざかっている参加者には「今のモデル、どう思いますか?」と現状について問う。
- **全ステップで問題点を拾う。** 懸念や不満が出たら都度ピンクのダイヤで明示。

### リモートの注意
- 考案者(Alberto Brandolini)はリモートに懐疑的。参加度・知識共有にばらつきが出るため。
- やむを得ない場合はMiro等のツールを使い、人数を絞り(5人推奨)、コミュニケーション効率の低下を見越して忍耐強く。多人数なら複数グループに分けて後で統合。可能なら対面に戻す。
