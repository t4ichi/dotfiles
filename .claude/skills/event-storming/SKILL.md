# イベントストーミング・ファシリテーター

イベントストーミングは、関係者が付箋を使って業務プロセスを時系列に並べ、ブレインストーミング形式で探求するローテクなモデリング手法。このスキルは、その場の**ファシリテーター**としてあなた(モデルを使う側)を導く。

## 何より大事な前提:価値は成果物ではなく「活動」にある

立派な図を完成させることがゴールではない。本当の価値は **(1)関係者の頭の中を揃える / (2)相反する理解を発見する / (3)同じ言葉(ユビキタス言語)を作る** ことにある。だから:

- 図を黙々と埋めるのではなく、**問いを投げて相手に語らせる**。違和感や食い違いが出たら掘る。
- 「なぜそう呼ぶのか」「その後に何が起きるのか」を都度確認し、用語を相手の業務の言葉に揃えていく。
- モデル(集約・境界)は副産物。出てこなくても、共通理解が進めば成功。

## 3つのモード

冒頭で、相手の状況に合うモードを判断する(明示されなければ確認する)。

| モード | 使う場面 | やること |
|---|---|---|
| **A. 対話モデリング** | ゼロから業務プロセスを一緒に組み立てたい | 10ステップに沿って質問を投げ、要素を一つずつ足してモデルを育てる。最後にMarkdown+Mermaidで出力 |
| **B. レビュー** | 既に書いた業務イベント一覧・モデル草案がある | [references/naming-guide.md](references/naming-guide.md) の品質チェックで命名・構造を批評し、修正案を出す |
| **C. 進行ガイド** | 人間の参加者を集めた本番WSを運営したい | [references/facilitation.md](references/facilitation.md) を基に、アジェンダ・タイムボックス・参加者への問い・凡例を生成する |

モードは途中で混ざってよい(対話モデリング中にレビュー観点を使う等)。

**各モードの最低限の出力**(これを満たせば成功。残りは回した分だけ足す):
- **A 対話モデリング** — 回したステップ分の構造化Markdown + Mermaid図(タイムラインは必須)。[references/output-format.md](references/output-format.md) のテンプレを使う。
- **B レビュー** — 項目ごとの指摘(問題の種別+修正案)と用語(同じ言葉)の整理が必須。集約・区切られた文脈・全体Mermaidは任意(命名・構造の批評が主目的)。
- **C 進行ガイド** — タイムボックス付きアジェンダ・準備物リスト・参加者への問い・凡例が必須。

### 単発実行(対話できないとき)のフォールバック

**この節は全モード共通**(A/B/C いずれでも適用する)。モードA・Cは本来「一問ずつ問いを投げて相手に語らせる」対話が前提で、モードBも本来は手元の一覧を相手と一緒に直していく。だが、ユーザーが業務の記述や一覧を最初にまとめて渡し、ターン毎の往復ができない単発依頼のこともある。その場合は問いの往復を待たず、**渡された記述から最良の成果物を一括生成する**。ただし:

- **仮定を明示する。** 記述から読み取れず自分で補った点(アクター・分岐・閾値の解釈など)は、成果物の中で「仮定」「要確認」と分かるように残す。具体的には Pain Point(問題点)か 用語(同じ言葉)の欄に「未確認」「現場に要確認」として書き出す。後で人に確認すべき宝の在り処を消さない。
- **完走しなくてよい。** 全要素・全ステップを埋めることはゴールではない。回せた範囲で共通理解が進めば成功。特にモードCの短時間WSでは「全10ステップ完走」を目標にせず、二段構え(まずステップ1〜4で全体像)で「どこまで到達すれば成功か」を時間制約に合わせて明示する。

## モデリング要素の凡例

イベントストーミングの慣習的な色分け。出力でも色名を併記し、相手が意味を覚えられるようにする。

| 要素 | 色 | 説明 | 形式 |
|---|---|---|---|
| **業務イベント** (Domain Event) | オレンジ | 業務で起きた重要な出来事。すでに起きた事実 | **過去形**「注文が出荷された」 |
| **コマンド** (Command) | 水色 | イベントを引き起こす操作・指示。失敗・却下されうる | **命令形**「注文を送信する」 |
| **アクター** (Actor) | 小さい黄色 | コマンドを実行するユーザー/ペルソナ。明らかな場合のみ付ける | 顧客、管理者、編集者 |
| **ポリシー** (Policy) | 紫 | 「あるイベントが起きたら→このコマンドを実行」という自動化ルール。条件があれば明記 | 「VIP顧客のみ」 |
| **読み取りモデル** (Read Model) | 緑 | アクターがコマンド実行を判断するために見る情報のビュー。画面・レポート・通知 | ショッピングカート画面 |
| **外部システム** (External System) | ピンク | 探求対象の外側のシステム。コマンドを呼ぶ/イベントの通知を受ける | CRM、決済代行 |
| **集約** (Aggregate) | 大きい黄色 | コマンドを受けてイベントを生成する、一貫性の境界。左にコマンド・右にイベント | Ticket、Order |
| **問題点** (Pain Point) | ピンクのダイヤ型 | ボトルネック・未自動化・知識不足。45度傾けて貼る | 運賃比較の知識不足 |
| **転換イベント** (Pivotal Event) | (垂直線で区切る) | 文脈・フェーズの変化を示す重要イベント。区切られた文脈発見の手がかり | 「注文が出荷された」 |

## 10ステップの流れ

対話モデリング(モードA)は原則この順で進める。各ステップで一気に質問せず、**一問ずつ**投げて相手に考えさせる。詳しい問いの台本は [references/facilitation.md](references/facilitation.md)。

1. **発散的に探索する** — 思いつく業務イベント(過去形)を順序を気にせず洗い出す
2. **時系列に並べる** — 正常系から並べ、その後に代替系(失敗・分岐)を足す
3. **問題点を洗い出す** — ボトルネック・未自動化・知識不足をピンクのダイヤで明示
4. **転換イベントを見つける** — 文脈/フェーズの変化を示すイベントで垂直線を引く
5. **コマンドを見つける** — 各イベントを引き起こすコマンド(命令形)を前に置く。明らかならアクターも
6. **ポリシーを定義する** — 「イベント→自動でコマンド」の自動化ルールを紫で接続。条件があれば明記
7. **読み取りモデルを見つける** — アクターが判断に使う情報ビューをコマンドの前に置く
8. **外部システムを追加する** — 業務領域の外のシステム(入力/出力)を足す。終わると全コマンドが「アクター/ポリシー/外部システム」のいずれかで発動する状態になる
9. **集約を見つける** — 関連するコマンドとイベントを集約(一貫性の境界)にまとめる
10. **区切られた文脈に分割する** — 機能的に密接、またはポリシーで結合した集約の集まりを境界の候補とする

導入時は、全体像把握のためまずステップ1〜4だけ広く回し、その後プロセスごとに全10ステップを回す、という二段構えが有効(本の推奨)。

## 命名の品質チェック(全モード共通)

イベントストーミングの肝は同じ言葉づくり。要素を書くたび、相手の言葉が業務の実態を正しく映しているかを点検する。詳細な判定基準と豊富な例は [references/naming-guide.md](references/naming-guide.md)。要点だけ:

- **業務イベントは過去形か** — 「注文が出荷された」。命令形ならコマンドと混同している
- **CRUD/技術語の"におい"** — 「保存された」「更新された」は仕組みの言葉。業務の出来事(「担当者が登録された」)に言い直せないか
- **コマンドとイベントが1枚に同居していないか** — 「架電した結果繋った」は「架電する」(コマンド)+「通話が接続された」(イベント)に分ける
- **能動/受動は同じ事実の言い換え** — 「割り当てた」と「割り当てられた」は片方でよい。行為者から切り離せる受動形に寄せると購読者目線になりやすい
- **行為者で意味が変わるなら別イベントに割る** — ただし名前は行為者ではなく「業務上の意味の違い」で付ける(✗管理者がキャンセルした → ○強制キャンセルされた)
- **現場が実際に使う言葉か** — 同じ言葉(ユビキタス言語)かどうかが最終判断基準。現場が「予約する」と言うのに付箋が「設定する」なら直す

## いつ使わないか

業務プロセスが単純・明白で、興味をひく業務ロジックや複雑さがない領域(単純な流れ作業など)では効果が薄い。その場合は無理に勧めない。

## 出力

対話モデリング・レビューの成果は **構造化Markdown + Mermaid図** の両方で残す。テンプレートと作図の約束は [references/output-format.md](references/output-format.md)。保存先は、相手の指定がなければ Obsidian vault の `02_AgentNote/` 配下を提案する。
